## 01 Kubernetes 整体架构（岗位核心基础）

### 1.1 整体组件调用链路

```plain text
kubectl（客户端）
       ↓（REST API）
kube-apiserver（集群唯一入口）
       ↓
┌──────────┬───────────┬────────────────┐
│   etcd   │ scheduler  │ controller-manager │
│(集群数据库)│(Pod调度器) │(控制器闭环管理)   │
└──────────┴───────────┴────────────────┘
       ↓
        kubelet（节点代理）
       ↓
     containerd（容器运行时）
       ↓
          Pod（最小调度单元）
```

### 1.2 控制平面 Control Plane（Master）

|核心组件|核心生产释义|
|---|---|
|**kube-apiserver**|集群唯一REST API入口；所有读写请求鉴权、限流、校验；集群组件通信中枢|
|**etcd**|集群唯一持久化存储；存储全量集群元数据、资源状态；强一致性KV数据库|
|**kube-scheduler**|监听未调度Pod；完成节点过滤+节点打分；决策Pod最终运行节点|
|**kube-controller-manager**|各类控制器合集；持续协调集群**实际状态→期望状态**，维持集群稳态|
|**cloud-controller-manager**|云环境专用；联动公有云API管理LB、云硬盘等云厂商底层资源（适配你的云环境栈）|

### 1.3 工作节点 Node

|核心组件|核心生产释义|
|---|---|
|**kubelet**|节点代理；全权管理本机Pod全生命周期：创建、启停、健康检查、资源上报|
|**containerd**|标准CRI容器运行时；镜像拉取、容器创建执行、容器生命周期管理|
|**kube-proxy**|节点网络代理；生成转发规则；实现Service集群内部流量转发|

### 1.4 RKE2 集群补充（你的技术栈重点）

RKE2 为标准化高可用K8s发行版；组件静态固化部署、自带证书自动轮换、简化控制面高可用搭建；生产环境多用于企业标准化集群交付。

## 02 K8s 核心设计思想（底层逻辑）

### 2.1 声明式 API（核心本质）

**核心定义**：用户只定义【业务期望最终状态】，K8s控制器自主完成闭环调谐，不关心执行过程。

业务场景举例：

1. 用户声明：需要3个运行Nginx的业务副本（`replicas: 3`）
2. 控制器轮询巡检：当前集群只有2个正常Pod
3. 控制器主动执行创建动作，补足副本至3个，达到稳态后停止操作

对比命令式：命令式关心**怎么做**；K8s声明式只关心**做成什么样**

## 03 Pod 原理与生命周期（高频核心知识点）

### 3.1 Pod 核心定位

K8s集群**最小调度、最小资源调度单元**；一个Pod内多个容器共享网络命名空间、存储挂载

### 3.2 标准 Pod 创建全流程（核心链路背诵）

```plain text
1. kubectl 发起创建请求 → 2. kube-apiserver 接收请求并参数校验
3. apiserver 写入资源数据至 etcd 持久化存储
4. Scheduler 监听apiserver资源变更，捕获未调度Pod
5. 调度器完成节点筛选+打分，绑定最优Node节点
6. 目标节点kubelet监听绑定任务，接收创建指令
7. kubelet调用containerd(CRI) 创建业务容器
8. CNI插件配置Pod网卡、集群网络
9. 容器启动就绪，更新Pod状态至etcd集群
```

## 04 工作负载：Deployment 原理（发布核心）

### 4.1 层级从属关系

```plain text
Deployment（上层控制器）
        ↓ 管理
ReplicaSet（副本控制器）
        ↓ 管理
Pod（业务运行实体）
```

### 4.2 Deployment 三大核心能力（生产刚需）

- **副本稳态管理**：保证集群常驻指定数量正常业务Pod
- **滚动更新发布**：版本迭代不停机上线
- **版本回滚**：线上故障一键回撤历史稳定版本

### 4.3 滚动更新底层逻辑

1. 存量版本：ReplicaSet(v1) 管控全部旧业务Pod
2. 触发升级：Deployment创建新ReplicaSet(v2)生成新版本Pod
3. 新旧副本交替扩容缩容，逐步替换全部业务实例

### 4.4 生产核心参数（岗位必备知识点）

- `maxSurge`：滚动更新最大超出期望副本数量
- `maxUnavailable`：滚动更新最大不可用副本数量

## 05 K8s 网络体系（你的技术栈强化重点）

## 5.1 K8s 底层网络三大铁律

1. 每个Pod独占独立集群IP，Pod内容器共享同一个IP
2. **所有Pod之间无NAT直接互通**
3. 节点IP与Pod网络通信无需转发NAT规则

## 5.2 Service 类型与生产适用场景

|Service类型|流量链路|生产适用场景|
|---|---|---|
|**ClusterIP**（默认）|业务Pod → ClusterIP → 后端Endpoint Pod|集群内部服务互通，业务微服务之间调用，无外部访问需求|
|**NodePort**|外网用户 → 节点IP:固定端口 → Service → 后端Pod|小规模集群临时对外暴露服务；不适合大规模生产业务|
|**LoadBalancer**（云环境主流）|外网用户 → 云厂商LB → Service → 后端Pod|公有云生产环境标准对外暴露方式；联动CCM自动编排云负载均衡（适配当前技术栈）|
|**ExternalName**|集群内Pod → 外部域名地址|集群内服务访问第三方外部域名、跨集群业务调度|

## 5.3 kube-proxy 三种转发模式（生产重点对比）

|转发模式|实现原理|生产优缺点|适用场景|
|---|---|---|---|
|**iptables**（默认）|节点内核生成iptables NAT规则，流量直接内核转发|优点：无需代理转发、性能适中；缺点：大规模集群规则爆炸，故障排查困难、无会话保持|中小规模常规集群|
|**IPVS**（生产推荐）|内核级负载均衡模块，独立负载均衡规则表|优点：高并发性能强、支持多种负载算法、规则易维护；缺点：内核参数需要调优|大规模生产集群、高并发业务（企业标准选型）|
|**userspace**（废弃）|kube-proxy用户层代理转发流量|性能极差、转发链路长，已经全线淘汰|无生产使用场景|

## 5.4 CNI 容器网络插件选型

|主流插件|核心特性|生产选型建议|
|---|---|---|
|**Flannel**|架构简单、部署零配置、默认VXLAN隧道网络；不支持网络策略|小型测试集群、非核心业务集群；运维成本最低|
|**Calico**（企业主流）|支持BGP路由/VXLAN；原生支持NetworkPolicy Pod网络隔离；成熟稳定|绝大多数企业生产集群；需要容器微服务流量隔离场景|
|**Cilium**（新一代高性能）|基于eBPF技术；内核级流量管控、可观测性强、高性能、L4/L7流量管控|高性能生产集群、云原生高阶集群、需要eBPF观测管控场景|

## 5.5 Ingress 流量网关

**核心作用**：解决多微服务统一入口问题，弥补Service无法实现七层HTTP/HTTPS路由、域名、路径转发短板，集群七层流量统一网关。

**流量架构对比**:

- 传统方案：多个独立NodePort/LoadBalancer，多套外网入口，运维杂乱
- Ingress方案：统一外网入口，根据域名/URL路径分发流量至后端不同Service

**主流 Ingress Controller 生产选型**:

1. **Nginx Ingress**：部署简单、生态成熟，绝大多数集群默认选型
2. **Traefik**：动态配置、自动证书管理、云原生友好，适合动态集群
3. **AWS/云厂商ALB Ingress**：公有云环境专用，对接云厂商应用层负载均衡

## 06 K8s 存储体系（云环境核心模块）

### 6.1 存储层级关系（生产核心链路）

```plain text
业务Pod（挂载使用）
        ↓ 绑定
PVC（存储声明，用户层面）
        ↓ 匹配/动态申请
PV（存储实体，集群层面）
        ↓ 对接
底层存储介质（云盘/NFS/CEPH/对象存储）
```

### 6.2 核心资源释义

- **Volume**：基础存储卷，生命周期跟随Pod；Pod删除数据丢失，适合临时存储
- **PVC**：用户发起存储资源申请，只定义存储大小、访问模式，不关心底层存储类型
- **PV**：集群预置的物理存储资源，由运维提前创建，供PVC绑定使用
- **StorageClass（生产必备）**：存储资源模板，**实现PV动态创建**；无需运维手动预制PV

**云环境生产流程**：Pod挂载PVC → PVC匹配对应StorageClass → SC调用云厂商接口自动创建云硬盘PV → 完成挂载；适配公有云自动扩缩存储

### 6.3 存储访问模式与回收策略

访问模式：ReadWriteOnce、ReadOnlyMany、ReadWriteMany；生产云盘大多使用RWO模式  
回收策略：Retain（保留）、Delete（删除）、Recycle（回收清理，基本废弃）

## 07 配置与敏感资源管理

### 7.1 ConfigMap

存储非敏感明文配置文件：业务yml配置、Nginx配置、环境变量；支持文件挂载、环境变量注入两种挂载方式

### 7.2 Secret

存储敏感加密数据：数据库密码、账号Token、SSL通信证书；集群内部Base64加密存储，不适合高强度机密数据

**生产规范**：禁止原生Secret存放核心机密数据；企业生产对接 ExternalSecret、HashiCorp Vault 外部机密管理组件，实现加密、权限管控、自动轮换

## 08 集群调度体系

### 8.1 Kube-Scheduler 完整调度流程

1. **预选阶段（过滤）**：筛选符合Pod运行条件节点；校验资源配额、节点标签、污点、Pod亲和性、PVC绑定状态
2. **优选阶段（打分）**：对筛选通过节点量化打分；依据剩余CPU/内存、节点负载、拓扑分布打分，选取最高分节点
3. **绑定阶段**：API Server完成Pod与目标节点绑定

## 8.2 生产常用调度管控能力

- **NodeSelector**：简单节点标签调度，强制Pod运行在带指定标签节点
- **NodeAffinity 节点亲和性**：高阶标签调度；分为硬性必须调度、软性优先调度
- **Pod亲和/反亲和**：控制Pod聚合部署或者隔离部署；多副本打散防单点故障核心能力
- **Taint(节点污点) + Toleration(Pod容忍)**：节点级别调度隔离；核心业务节点禁止普通业务调度、专用节点隔离业务

## 09 容器健康检测（探针）

三大内置探针，实现业务自愈、流量熔断、慢启动保护，生产业务必须配置

|探针类型|核心作用|探测失败集群动作|
|---|---|---|
|**LivenessProbe 存活探针**|检测容器进程是否正常存活|直接重启异常容器，实现业务自愈|
|**ReadinessProbe 就绪探针**|检测业务是否可正常接收流量|剔除对应Endpoint，切断流量流入，不重启容器|
|**StartupProbe 启动探针**|针对启动缓慢的老旧应用，延长初始化检测时间|启动超时直接重启容器；保护慢启动业务不被误杀|

## 10 容器资源配额管理

### 10.1 Requests / Limits 核心区别

- **Requests（申请资源）**：调度器调度依据；集群保证该额度资源可用，Pod最低保障资源
- **Limits（上限资源）**：容器资源硬限制；超出配额CPU限流、内存OOM杀，防止业务抢占节点资源

### 10.2 生产资源单位规范

- CPU：1000m = 1物理核心；小数直接填写0.5
- 内存：Mi/Gi（二进制标准，生产推荐）；MB/GB（十进制，不推荐）

## 11 业务自动扩缩容 HPA

实现业务根据负载自动增减Pod副本，应对流量波峰波谷，节约集群资源

- 依赖组件：**metrics-server**（集群资源指标采集基础组件）
- 基础指标：CPU使用率、内存使用率
- 高阶指标：自定义业务QPS、接口延时（Prometheus适配器）
- 运行逻辑：监控指标触发阈值 → 自动扩容；流量回落自动缩容

## 13 生产发布策略

- **滚动更新（RollingUpdate，默认）**：不停机灰度分批发布；业务无中断，绝大多数业务默认方案
- **蓝绿发布（Blue/Green）**：两套独立完整业务环境；全量切换流量；故障秒级回滚，适合核心金融业务
- **金丝雀灰度发布（Canary）**：小比例流量放行新版本，验证业务稳定性后全量放量
  
**高阶生产工具**：复杂灰度流程使用 Argo Rollouts、Istio 服务网格实现精细化流量管控

## 14 监控告警体系（你的核心技术栈）

### 14.1 整体架构：Prometheus + Thanos + Grafana

```plain text
集群指标采集端 → Prometheus（本地时序存储+告警）
        ↓
Thanos（分布式聚合、长期存储、多集群汇总）
        ↓
Alertmanager（告警收敛、分级推送）
        ↓
Grafana（可视化大盘、报表展示）
```

### 14.2 核心采集组件

- node-exporter：服务器硬件、操作系统底层指标采集
- kube-state-metrics：K8s集群各类资源对象状态指标采集
- kubelet内置metrics：节点容器运行、网络资源指标

### 14.3 Thanos 核心生产价值

解决Prometheus单集群、存储容量不足、数据无法跨集群汇总问题；实现监控数据长期归档、多集群统一监控视图

## 15 日志收集架构（你的核心技术栈）

### 15.1 生产标准链路：FluentBit + Kafka + ES + Loki

```plain text
容器标准控制台日志stdout
        ↓
Fluent Bit（节点轻量日志采集，生产首选）
        ↓
Kafka（消息队列缓冲）
        ↓
Elasticsearch / Loki（日志持久化存储）
        ↓
Kibana / Grafana（日志检索、可视化、审计）
```

### 15.2 链路引入Kafka生产原因

- 流量削峰：业务日志暴涨时缓冲流量，避免打垮后端存储
- 服务解耦：采集端和存储端互不依赖，组件独立扩容
- 数据缓冲：后端存储故障时日志不丢失，故障恢复后消费补发

## 16 生产故障排查流程（核心实操能力）

### 16.1 Pod 无法正常启动通用排查流程

1. 优先执行：资源事件排查 `kubectl describe pod` 查看原生事件报错（最快定位原因）
2. 容器运行异常：查看业务运行日志 `kubectl logs / kubectl logs --previous`

### 16.2 高频异常场景&根因定位

#### 16.2.1 Pod 状态 Pending（调度异常）

常见根因：PVC未绑定/存储不存在、集群资源配额不足、节点污点拒绝调度、Pod亲和性规则不满足、集群IP地址池耗尽

#### 16.2.2 Pod 状态 CrashLoopBackOff（容器反复崩溃重启）

常见根因：启动脚本/配置文件错误、业务程序代码异常、健康探针配置不合理、内存溢出OOM、启动权限不足；优先查看历史容器日志--previous

#### 16.2.3 Service 服务网络不通

标准排查链路：Pod业务正常性 → Endpoint后端健康实例是否存在 → Service规则配置 → kube-proxy运行状态 → 内核iptables/IPVS转发规则 → 底层CNI网络连通性

## 17 K8s 集群安全体系（CKS 能力方向）

### 17.1 RBAC 权限管控（集群权限核心）

集群资源访问权限体系；管控用户/服务账号集群操作权限：User/ServiceAccount → Role/ClusterRole → RoleBinding/ClusterRoleBinding，最小权限原则管控集群操作

### 17.2 NetworkPolicy 容器网络隔离

Namespace、Pod级别的东西向流量隔离；限制Pod之间、跨命名空间网络访问，防范集群内部横向渗透攻击

### 17.3 敏感资源安全规范

禁止YAML明文挂载密码、证书；禁用原生Secret存储核心机密；对接外部机密管理系统；密钥定期轮换

## 18 集群高阶生产运维能力

### 18.1 K8s 集群版本升级流程（RKE2集群）

1. 前置操作：全量etcd集群数据备份、核心业务快照备份
2. 控制面升级：分批升级Master节点控制平面组件，保障集群高可用不中断
3. 工作节点升级：节点排水腾空业务、升级kubelet/容器运行时、业务平滑迁移
4. 收尾：全集群组件状态校验、业务全链路功能回归验证

### 18.2 集群高可用保障方案

- 控制面高可用：3节点Master集群、etcd集群多数派quorum机制保证数据一致性
- 入口高可用：API Server前置负载均衡，消除单点控制面入口
- 业务高可用：多副本打散部署、多可用区部署、故障自动迁移

### 18.3 大规模集群性能优化方向

- etcd性能调优：磁盘IO、碎片清理、QPS限流、数据备份策略
- API Server QPS限流与缓存优化，规避大集群接口卡顿
- kubelet节点参数调优、节点最大Pod数量限制
- 集群网络规模优化、大集群IPVS参数调优
- 监控系统容量规划、海量指标存储优化（Thanos核心优化场景）

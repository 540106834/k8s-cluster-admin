# GitOps 完整架构与工作机制（精简版）

## 一、GitOps 核心定义

GitOps 是面向 Kubernetes 的**声明式持续交付模型**，核心准则为 **Git 唯一可信源**。所有集群资源配置统一托管在 Git 仓库，禁止集群手动修改配置，通过控制器自动比对、同步集群状态，实现标准化、可追溯、可自愈的自动化交付。

## 二、核心架构分工（CI + GitOps 解耦）

### 1. CI 层（只构建、不部署）

负责代码编译、测试、镜像构建、镜像推送，仅修改 Git 配置仓库中的资源清单（更新镜像版本、配置参数），不直接操作 Kubernetes 集群，最小化集群权限风险。  
主流工具：GitLab CI、Jenkins、GitHub Actions

### 2. Git 配置层（唯一配置源）

采用**双仓库规范**，实现代码与配置完全解耦：

- **业务仓库**：存储业务代码、Dockerfile，仅用于构建镜像
- **配置仓库**：存储所有 K8s 资源 YAML（Deployment、Service、Ingress、ConfigMap 等），作为集群部署的唯一标准

### 3. GitOps 控制器层（只部署、不构建）

核心组件：ArgoCD、Flux  
常驻集群运行，持续监听 Git 配置仓库变更，自动完成状态比对与集群同步，接管所有 CD 交付工作。

### 4. 集群运行层

Kubernetes 集群仅作为最终运行载体，所有资源状态由 Git 驱动，无人工临时配置变更。

## 三、GitOps 完整全流程（详细落地拆解）

GitOps 整体流程严格遵循 **代码变更→CI构建→配置变更→控制器监听→状态比对→集群同步→自愈校验** 的闭环链路，全程无人工干预集群操作，每一步均有明确的组件分工与执行逻辑，以下为逐阶段详细拆解。

### 阶段1：业务代码变更与CI构建（纯代码层，不触碰集群）

开发人员完成业务代码迭代、Bug修复、版本更新后，向业务代码仓库提交代码并合并至主干分支。代码仓库的 WebHook 触发器自动触发 CI 流水线（GitLab CI/Jenkins）启动执行。  
流水线依次完成标准化质量与构建流程：首先执行代码静态扫描、单元测试、漏洞检测，拦截不合格代码；随后根据 Dockerfile 构建业务镜像，自动打上版本标签（时间戳/CommitID/语义化版本）；最后将构建完成的安全镜像推送至企业私有镜像仓库（Harbor），完成镜像归档与版本留存。

本阶段核心特点：**仅处理业务代码与镜像，不连接、不操作K8s集群，无集群权限暴露风险**。

### 阶段2：CI自动更新Git配置仓库（唯一可信源变更）

镜像构建推送成功后，CI 流水线不会直接部署应用，而是通过自动化脚本（yq/kustomize/helm）修改**独立的GitOps配置仓库**中的K8s资源清单。  
核心修改内容为 Deployment 资源的 `image` 镜像版本字段，同步更新配置变更、环境变量、资源配额、探针参数等运维配置，修改完成后自动提交、推送至配置仓库，生成唯一 Commit 记录。  
通过双仓库隔离，实现**开发只管代码迭代，部署变更统一由配置仓库管控**，所有上线、配更操作全部留痕可追溯。

### 阶段3：GitOps控制器监听仓库变更

集群内常驻的 ArgoCD/Flux 控制器，通过长轮询+WebHook双重机制，持续监听Git配置仓库的分支变更。一旦检测到新的Commit提交、YAML配置变更，控制器立即触发应用同步流程。  
控制器会首先拉取配置仓库最新的YAML清单，解析当前应用的**期望状态**，包含：镜像版本、副本数量、资源限制、网络策略、挂载配置、探针规则等全量集群资源定义。

### 阶段4：核心状态Diff比对（GitOps核心机制）

控制器调取Kubernetes APIServer，查询当前集群中对应应用的**实际运行状态**，与Git仓库中的期望状态进行全维度Diff比对。  
比对范围包含：镜像版本差异、副本数偏差、配置文件变更、标签注解差异、资源配额变更、探针参数变动、网络权限变更等所有资源字段。只要存在任意状态不一致，即判定为需要同步更新。  
该机制彻底解决传统CICD状态脱节问题，保证集群状态永远贴合配置定义。

### 阶段5：自动化集群同步与滚动更新

状态比对出现差异后，控制器根据应用同步策略，自动执行配置同步操作，将Git中的标准YAML配置应用到K8s集群。  
针对无状态应用（Deployment），控制器触发标准滚动更新机制：逐步创建新版本Pod，就绪后销毁旧版本Pod，更新过程中保证业务不中断、零停机。针对有状态应用（StatefulSet）、配置类资源，严格按照K8s原生更新逻辑执行，保证数据与服务稳定性。

### 阶段6：上线健康校验与状态同步确认

更新完成后，控制器联动K8s就绪探针、存活探针、应用健康接口，持续校验新版本Pod运行状态。待所有副本就绪、服务流量正常、无报错日志后，标记应用同步状态为 **Healthy**。  
若更新过程中出现启动失败、探针超时、服务异常，控制器会自动终止更新，保留旧版本服务运行，避免业务中断，并记录异常日志用于问题排查。

### 阶段7：持续自愈与一致性守护（生产核心能力）

应用稳定运行后，控制器不会终止流程，会持续循环执行状态比对。如果运维人员手动在集群内修改Pod、配置、副本数等资源，导致集群实际状态与Git期望状态不一致，**自愈机制会自动强制还原集群配置**。  
该能力从机制上杜绝集群临时篡改、配置漂移问题，永久保障「Git定义什么，集群就运行什么」的一致性原则。

### 阶段8：版本回滚与变更审计闭环

若上线后出现业务异常、性能问题，无需重新打包构建，仅需要在Git配置仓库回滚对应Commit。控制器检测到配置回滚变更后，会自动同步旧版本配置，完成应用秒级回滚，快速恢复业务。  
所有变更记录、上线记录、回滚记录全部留存Git日志与控制器日志，全链路可审计、可追溯，满足生产环境合规运维要求。

```Plain Text
完整细化闭环链路
开发提交代码 → CI质检/构建/推送镜像 → CI更新Git配置仓库YAML
→ ArgoCD/Flux监听变更拉取期望状态 → 比对集群实际状态Diff
→ 自动同步配置、滚动更新集群 → 健康探针校验上线结果
→ 持续循环巡检、自动修复配置漂移 → 全流程日志留存、支持一键回滚
```

## 四、控制器核心工作机制

### 1. 状态比对机制（核心）

控制器循环执行 **Git 期望状态 VS 集群实际状态** Diff 比对，识别配置、版本、副本数等所有差异，是 GitOps 自动化的核心依据。

### 2. 三种同步模式

- **手动同步**：控制台手动触发 Sync，按需更新集群配置
- **自动同步**：检测到 Git 变更后，立即自动同步至集群
- **自愈同步（生产核心）**：集群内任何手动修改资源的操作，会被控制器自动还原，严格保证集群与 Git 配置一致

## 五、核心特性与落地优势

- **全链路可追溯**：所有配置变更均留存 Git Commit 记录，便于审计、排查问题
- **快速回滚**：业务异常时，仅需回滚 Git 提交，即可实现集群配置秒级回滚
- **权限安全可控**：CI 无需集群高权限，规避流水线直接操作集群的安全风险
- **多集群统一管理**：一套 Git 配置可同步至多个集群，适配大规模运维场景
- **标准化交付**：规避人工操作差异，统一部署规范，适配团队协作与持续迭代

## 六、GitOps 与传统 CICD 核心差异

**传统 CICD**：流水线直接通过 kubectl 操作集群，配置与 Git 脱节，易出现状态不一致、变更不可追溯、多集群管理混乱等问题。  
**GitOps**：流水线只改 Git，控制器统一部署，以配置驱动集群，实现交付标准化、状态一致性、运维可控化。

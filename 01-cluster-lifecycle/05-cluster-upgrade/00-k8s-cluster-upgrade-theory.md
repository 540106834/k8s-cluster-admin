# Cluster Upgrade 集群升级理论认知手册（含自建K8s + 云厂商托管K8s）
## 一、核心定义：什么是Kubernetes集群升级
Kubernetes 集群升级是**控制平面 + 节点运行时组件**的版本迭代行为，本质是替换集群二进制/容器镜像、同步API语义、更新存储层（etcd）、适配网络/监控插件，解决三类核心诉求：
1. 安全：修复内核、apiserver、kubelet高危漏洞
2. 功能：使用新版本API、调度、存储、网络新特性
3. 兼容：满足业务、中间件、第三方组件新版本依赖

升级分为两大体系：
1. **自建集群升级**（kubeadm/二进制部署，你当前1.32.13单Master集群）
2. **云厂商托管K8s升级**（EKS/ACK/GKE/AKS等云托管集群，云厂商接管控制平面）

## 二、K8s版本体系理论（升级底层规则）
### 2.1 版本语义规则 `vX.Y.Z`
- X：Major 大版本（跨代升级，破坏性变更）
- Y：Minor 小版本（功能迭代，API新增/废弃，半兼容）
- Z：Patch 补丁版本（漏洞修复，无API破坏性变更，平滑滚动）
示例：`v1.32.13`
- Major=1（K8s主线统一1）
- Minor=32（功能迭代分支，1.32 → 1.33 属于跨Minor大版本升级）
- Patch=13（安全补丁，1.32.13 → 1.32.14 补丁平滑升级）

### 2.2 官方版本支持周期规则（升级强制依据）
1. 每个Minor版本官方维护 **12个月**，到期停止安全补丁；
2. 同一时间官方仅维护最近3个Minor分支（如1.31/1.32/1.33）；
3. 过期版本存在无补丁高危漏洞，必须完成升级；
4. 版本升级只能**顺序递进**：
   - 补丁升级：1.32.13 → 1.32.14（无限制，随时滚动）
   - Minor跨代：1.32 → 1.33，禁止跳过 1.32 直接升到1.34。

### 2.3 集群组件版本对齐理论
集群所有核心组件必须版本匹配，版本差不允许超过1个Minor：
1. 控制平面（apiserver/controller/scheduler/etcd/coredns）版本最高；
2. kubelet 版本允许比apiserver低最多1个Minor；
3. kubectl 允许比集群高/低1个Minor；
4. 第三方组件（Calico/metrics-server）必须匹配对应K8s Minor版本，否则API调用报错。

## 三、两类升级核心分类：Patch补丁升级 / Minor跨代升级
### 3.1 Patch补丁升级（同Minor，如1.32.13→1.32.14）
#### 理论特征
1. 无API废弃、无存储格式变更、无配置字段删除；
2. 控制平面、节点滚动更新，业务几乎无感知；
3. 回滚成本极低，降级组件即可恢复；
4. 第三方CNI/监控组件无需同步升级。
#### 适用场景
漏洞修复、稳定性bug修复、小功能优化。

### 3.2 Minor跨代大版本升级（1.32 → 1.33）
#### 理论风险点（核心认知）
1. **API废弃机制**
   K8s每个Minor会标记部分API为`deprecated`，2个Minor后彻底移除；升级后旧yaml直接报错。
2. **etcd存储兼容变更**
   Minor迭代会调整数据序列化、索引机制，etcd快照只能同版本完整恢复，跨版本快照无法兼容。
3. **运行时配置废弃**
   containerd/kubelet配置字段、命令行参数被删除，旧配置直接导致组件启动失败。
4. **CRD/Operator不兼容**
   自研控制器、中间件CRD依赖旧API组，升级后无法创建资源。
5. **网络插件不兼容**
   Calico、Istio等组件必须同步升级对应适配版本，否则节点NotReady、Pod不通。
#### 运维约束
必须全量备份资源、etcd双快照、业务低峰维护窗口、测试集群预演。

## 四、自建kubeadm集群升级完整理论模型
### 4.1 集群分层升级顺序（不可逆，底层原理）
控制平面先行 → 工作节点分批 → 第三方插件
1. Master控制平面：kubeadm升级 → apiserver/etcd静态Pod滚动更新
   原理：apiserver是集群唯一入口，必须先完成新版本提供新API，节点才能适配。
2. Master本地kubelet/kubectl同步升级
3. Worker节点**串行分批升级**
   原理：同时驱逐所有节点会导致业务完全中断；单台drain调度至剩余节点，保证业务高可用。
4. CNI、metrics-server等集群附加组件滚动更新

### 4.2 自建集群升级依赖体系
1. 软件源：`kubernetes-new` 分版本APT源，区分不同Minor分支；
2. 镜像依赖：所有系统镜像（apiserver/etcd/pause）必须预上传内网Harbor，离线环境无外网拉取；
3. 存储依赖：etcd持久化磁盘SSD，升级前双快照备份；
4. 运行时依赖：containerd/runc版本必须匹配目标K8s版本。

### 4.3 自建集群回滚底层逻辑
1. Patch升级：仅降级kubeadm/kubelet/kubectl组件，重启kubelet即可恢复；
2. Minor跨代故障：
   - 轻度：降级组件，重新执行旧版本`kubeadm upgrade apply`修复控制平面配置；
   - 重度：静态Pod全部停止，使用升级前etcd快照覆盖数据目录，完整回滚集群状态。

## 五、云厂商托管K8s（EKS/ACK/GKE/AKS）升级理论
### 5.1 云托管集群与自建集群核心差异
| 维度 | 自建kubeadm集群 | 云厂商托管K8s |
|------|----------------|---------------|
| 控制平面归属 | 用户自行运维Master/etcd | 云厂商托管，用户无Master节点权限 |
| 升级操作主体 | 用户手动执行kubeadm | 控制台/API提交升级任务，厂商后台灰度升级控制平面 |
| etcd运维 | 用户自行备份、扩容、故障恢复 | 厂商三副本高可用，自动快照，用户仅可导出备份 |
| 系统镜像 | 用户自行上传Harbor | 厂商内置公共镜像仓库，无需自行维护系统镜像 |
| 故障兜底 | 用户手动回滚etcd快照 | 厂商底层自动回滚控制平面，用户仅需处理节点池 |
| 升级约束 | 用户自主控制时间窗口 | 厂商限制升级跨度，禁止跨2个Minor，强制递进升级 |

### 5.2 云托管集群两层分离架构（升级核心理论）
云K8s分为两层，**升级完全解耦**：
1. **托管控制平面（Master层，厂商维护）**
   - apiserver/etcd/controller/scheduler由云厂商多副本托管；
   - 用户无法登录Master节点，不能执行kubeadm；
   - 升级动作：用户提交版本升级工单，厂商后台滚动更新控制平面，全程无用户介入。
2. **节点池NodeGroup（用户自有节点）**
   - 云服务器ECS/虚拟机，kubelet、containerd由用户/节点镜像管理；
   - 控制平面升级完成后，用户分批滚动升级节点池节点；
   - 节点池支持：原地升级、新建节点灰度替换两种模式。

### 5.3 云K8s两种节点升级模式理论
#### 模式1：原地滚动升级（在线升级）
云平台批量分批重启节点，升级kubelet二进制，分批drain Pod，不新增机器；
优点：不额外产生云服务器成本；
缺点：节点升级期间业务Pod漂移，短时抖动。

#### 模式2：灰度替换节点池（生产推荐）
1. 新建同配置新版本节点池（匹配升级后K8s版本）；
2. 业务Pod逐步调度至新节点；
3. 排空旧节点池机器，删除旧节点；
优点：故障可快速切回旧节点池，风险极低；
缺点：升级期间双倍云服务器成本。

### 5.4 云托管集群升级限制规则
1. 控制平面只能逐Minor升级，例如1.32 → 1.33，无法直接1.32→1.34；
2. 控制平面升级完成前，不允许升级节点池；
3. 节点kubelet版本不能高于控制平面Minor版本；
4. 厂商会提前30天推送版本过期提醒，过期集群不再提供技术支持；
5. 部分厂商强制锁定弃用API资源，升级前平台会预检yaml废弃字段。

### 5.5 云K8s升级责任边界（运维核心认知）
1. **厂商负责**：控制平面升级、etcd高可用、Master故障恢复、系统镜像维护、底层安全补丁；
2. **用户负责**：
   - 业务yaml废弃API改造；
   - 节点池kubelet/containerd版本升级；
   - CNI、metrics、Ingress、中间件Operator版本适配；
   - 业务应用兼容性测试、业务数据备份。

### 5.6 云集群升级回滚机制
1. 控制平面：升级过程中厂商自动灰度，出现异常自动回滚至上一版本，用户无操作；
2. 节点池原地升级故障：单节点故障自动跳过，完成后单独重建节点；
3. 灰度节点池方案回滚：直接调度Pod回旧节点池，删除新节点池即可快速回退。

## 六、升级通用底层风险理论（自建/云集群通用）
### 6.1 API兼容性风险（最高发）
Minor版本持续废弃v1beta1类API，升级后Deployment/Ingress/CRD无法创建；
解决方案：升级前全局扫描集群资源，替换为稳定v1 API。

### 6.2 镜像拉取风险
自建集群：Harbor缺失新版本系统镜像，Pod ImagePullBackOff；
云集群：私有镜像仓库权限、内网连通性故障导致节点拉取失败。

### 6.3 网络连通风险
CNI插件版本与K8s不匹配，calico-node崩溃，节点长期NotReady，Pod失联。

### 6.4 存储层etcd风险
跨Minor升级etcd数据格式变更，无备份则集群数据彻底丢失；云集群虽厂商备份，仍建议用户额外导出资源yaml。

### 6.5 运行时不兼容风险
containerd/runc版本过低，新版本kubelet无法创建容器，节点NotReady。

### 6.6 业务调度中断风险
批量驱逐节点、污点策略、资源Limit缺失导致业务OOM、服务不可用。

## 七、标准化升级前置验证理论（通用流程）
无论自建还是云集群，升级前必须完成四层校验：
1. **集群健康层**
   节点全部Ready、系统Pod无异常、控制平面组件健康、无报错事件；
2. **API兼容层**
   扫描全命名空间资源，清理废弃API、废弃字段；
3. **备份层**
   自建：双份etcd快照 + 全资源yaml导出；
   云：导出全部业务资源yaml，开启厂商etcd自动备份；
4. **组件兼容层**
   CNI、监控、Ingress、Operator确认存在适配新版本的发布版本，镜像提前准备。

## 八、自建集群 vs 云集群升级核心对比总结
1. **自建集群**
   完全自主可控，升级流程长、操作步骤多；风险自担，需要运维精通etcd、kubeadm、containerd；适合离线机房、私有数据中心。
2. **云托管K8s**
   屏蔽控制平面复杂运维，升级操作简化；厂商兜底Master高可用与etcd；成本更高，依赖云厂商API与镜像服务；适合互联网业务、快速迭代场景。

## 九、企业升级规范理论（落地标准）
1. 补丁升级：月度执行，修复安全漏洞，无需业务停机窗口；
2. Minor跨代大版本升级：每6~8个月规划一次，提前搭建复刻测试集群完整预演；
3. 分级发布：测试集群升级验证 → 预发集群灰度 → 生产低峰执行；
4. 最小权限：升级操作仅管理员执行，操作全程留审计日志；
5. 备份留存：升级备份至少保留7天，观察业务稳定后清理。

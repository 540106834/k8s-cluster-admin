# ETCD 故障恢复实操规范（13\-etcd\-restore\.md）Ubuntu22\.04 离线生产版

## 1\. 文档概述

### 1\.1 文档定位

本文为 Kubernetes 生产集群 **ETCD 故障专项恢复、数据回滚、集群重建、异常修复** 标准化实操文档，适配 **Ubuntu22\.04、kubeadm 自建离线 Harbor 集群、K8s v1\.32**。作为 `12-etcd-backup.md` 配套恢复手册，专注故障落地修复，覆盖单节点/多节点ETCD损坏、数据误删、快照回滚、集群瘫痪急救全流程，是生产ETCD灾难性故障唯一恢复基准文档。

### 1\.2 核心恢复原则

- **先止损后恢复**：故障发生即刻暂停所有发布、变更、运维操作，避免数据二次损坏

- **快照优先回滚**：所有数据异常、组件损坏场景，优先使用有效一致性快照恢复

- **单节点兜底重建**：HA集群故障统一采用「单节点恢复→重新扩集群」标准方案

- **恢复必校验**：恢复完成强制校验ETCD健康、集群状态、业务完整性

- **故障留痕**：保留损坏原始数据，用于事后复盘溯源

### 1\.3 适用故障场景

- 人为误删Namespace、核心资源、配置导致集群异常

- ETCD单节点/多节点数据损坏、磁盘异常、文件丢失

- 集群升级、证书续期、配置变更导致ETCD数据错乱

- ETCD集群分裂、节点离线、状态不一致

- 集群彻底瘫痪、kubectl无法操作、所有组件异常场景

## 2\. 恢复前置规范（生产强制）

### 2\.1 故障现场保护

- 立即停止CI/CD流水线、手动发布、节点运维、资源变更操作

- 暂停所有Master节点kubelet自动重启，锁定故障现场

- 备份当前损坏的ETCD数据目录，留存故障原始数据用于复盘

### 2\.2 有效快照筛选标准

仅满足以下条件的快照可用于生产恢复，禁止使用无效快照：

- 通过 `etcdctl snapshot status` 校验完整无报错

- 快照时间点早于故障发生时间、集群状态稳定

- 快照文件大小正常、无空文件、无损坏

- 优先选择**变更前手动备份快照**，其次选择定时自动快照

### 2\.3 统一环境变量

所有恢复节点统一配置ETCD环境变量，保证证书与API版本一致：

```bash
export ETCDCTL_API=3
export ETCDCTL_CACERT=/etc/kubernetes/pki/etcd/ca.crt
export ETCDCTL_CERT=/etc/kubernetes/pki/etcd/peer.crt
export ETCDCTL_KEY=/etc/kubernetes/pki/etcd/peer.key
```

## 3\. 通用基础恢复流程（单Master集群）

适用于单Master集群ETCD数据损坏、资源误删、集群异常场景，流程简单、落地性强。

### 3\.1 停止集群核心服务

```bash
# 停止kubelet，终止所有集群组件运行
systemctl stop kubelet

# 确认ETCD容器完全停止
crictl ps | grep etcd | awk '{print $1}' | xargs crictl stop

```

### 3\.2 备份损坏旧数据

```bash
# 备份故障数据目录
mv /var/lib/etcd /var/lib/etcd-error-bak-$(date +%Y%m%d)

# 新建空数据目录
mkdir -p /var/lib/etcd

```

### 3\.3 执行快照恢复

```bash
# 替换为本地有效快照路径
etcdctl snapshot restore /data/etcd-backup/有效快照文件.db \
--data-dir=/var/lib/etcd

```

### 3\.4 重启服务恢复集群

```bash
systemctl start kubelet

```

### 3\.5 基础状态校验

```bash
# 查看节点状态
kubectl get nodes

# 查看系统组件状态
kubectl get pods -n kube-system

```

## 4\. 高可用集群标准恢复流程（3节点ETCD）

**生产HA集群核心恢复规则**：多节点ETCD故障、数据不一致、集群分裂，禁止逐节点单独恢复，统一采用「**单节点恢复→清空其余节点→重新扩集群**」标准方案，彻底解决集群数据不一致问题。

### 4\.1 全节点停止服务

所有Master节点依次执行，停止kubelet与ETCD服务

```bash
systemctl stop kubelet
crictl ps | grep etcd | awk '{print $1}' | xargs crictl stop

```

### 4\.2 选定主恢复节点

- 选择状态相对正常、磁盘无损坏的Master节点作为**主恢复节点**

- 其余所有Master节点清空ETCD故障数据目录

```bash
# 备节点执行：清空故障数据
mv /var/lib/etcd /var/lib/etcd-broken-bak-$(date +%Y%m%d)
mkdir -p /var/lib/etcd

```

### 4\.3 主节点快照恢复

仅在主恢复节点执行快照恢复操作

```bash
etcdctl snapshot restore /data/etcd-backup/有效快照.db --data-dir=/var/lib/etcd

```

### 4\.4 单节点启动集群

主恢复节点启动kubelet，仅保留单节点ETCD集群正常运行

```bash
systemctl start kubelet

```

### 4\.5 重新扩容ETCD集群

单节点集群恢复正常后，逐台将其余Master节点加入ETCD集群，完成HA重建，彻底解决集群分裂、数据不一致问题。

## 5\. 细分故障场景专项恢复方案

### 5\.1 资源误删恢复（Namespace/Pod/配置）

故障特征：节点正常、ETCD进程正常，业务资源丢失、配置错乱

恢复方案：无需停机重建，选择**故障前最新快照**执行增量回滚，恢复后校验资源完整性，核对业务数据。

### 5\.2 单ETCD节点宕机损坏

故障特征：HA集群单节点ETCD异常、离线，其余节点正常，集群可用但容错降级

恢复方案：异常节点清空故障数据，重启服务自动同步集群数据，无需全集群恢复。

### 5\.3 多ETCD节点故障/集群分裂

故障特征：多节点ETCD离线、集群成员不一致、数据同步失败、集群瘫痪

恢复方案：严格执行本文**4\.高可用集群标准恢复流程**，单节点重建后扩容HA集群。

### 5\.4 ETCD磁盘只读/文件损坏

故障特征：磁盘异常、ETCD日志报错、数据写入失败、进程反复重启

恢复方案：修复磁盘故障，清空损坏数据目录，通过快照重建节点数据。

## 8\. 生产红线禁止规范

- 禁止HA多节点ETCD故障时逐节点单独恢复，导致集群数据分裂加重

- 禁止使用未校验、损坏、过期快照执行生产恢复

- 禁止故障未止损、变更未暂停时执行恢复操作

- 禁止恢复后不做全维度验收，直接交付业务运行

- 禁止覆盖删除故障原始数据，不留复盘依据

- 禁止集群未完全稳定、ETCD未健康时重启业务发布

## 9\. 常见恢复报错与排错方案

### 9\.1 快照恢复后ETCD启动失败

常见原因：数据目录权限异常、证书不匹配、快照与集群版本不兼容。解决方案：修复目录权限、核对ETCD环境变量、使用同版本有效快照。

### 9\.2 恢复后集群节点状态不一致

常见原因：HA集群多节点未统一恢复、残留旧数据。解决方案：清空所有备节点数据，重新扩容集群。

### 9\.3 恢复后资源缺失/错乱

常见原因：快照时间点晚于故障点、快照本身无效。解决方案：更换更早的稳定快照重新回滚。

### 9\.4 ETCD集群健康检测超时

常见原因：网络端口不通、防火墙未关闭、节点时间不同步。解决方案：校验节点基线、时间同步、防火墙状态。

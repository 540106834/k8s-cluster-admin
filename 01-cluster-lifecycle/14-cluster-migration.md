# Kubernetes 集群平滑迁移割接规范（14\-cluster\-migration\.md）Ubuntu22\.04 离线生产版

## 1\. 文档概述

### 1\.1 文档定位

本文为 Kubernetes 生产集群**整机迁移、环境搬迁、版本迭代割接、机房迁移、新旧集群替换**标准化运维规范，完全适配 **Ubuntu22\.04、K8s v1\.32、kubeadm 自建离线 Harbor 集群**。统一离线环境下集群迁移全流程标准，覆盖新集群基线搭建、资源批量迁移、数据同步、流量灰度割接、旧集群下线全闭环流程，实现**业务零中断、数据零丢失、迁移可回滚**生产目标，为集群迁移唯一落地基准文档。

### 1\.2 核心迁移原则（生产红线）

- **新旧双集群并行**：迁移全程保留旧集群稳态运行，禁止先销毁旧集群再迁移，兜底回滚

- **数据优先同步**：配置、存储、密钥、状态数据全量迁移，保证新旧集群数据一致性

- **灰度流量割接**：先迁移配置、再拉起业务、最后灰度切流量，杜绝一刀切切换

- **离线闭环迁移**：全程不依赖外网，镜像、资源、配置全部依托内网Harbor离线流转

- **全流程可回滚**：每一步迁移操作均保留备份、留存快照，异常即刻回退旧集群

- **迁移必验收**：资源、服务、数据、流量、监控全维度验收后，方可下线旧集群

### 1\.3 适用场景

- 机房搬迁、服务器硬件迭代、老旧设备替换

- 旧版本K8s集群整体迁移、版本统一迭代升级（跨大版本无法原地升级场景）

- 集群环境基线错乱、组件异常，需重建标准化集群并迁移业务

- 测试/预发环境复刻、生产环境克隆、集群扩容重构

- 单节点集群迁移至高可用多Master集群，提升集群容灾能力

## 2\. 迁移前置规划与准备

### 2\.1 新旧集群基线对齐

迁移前必须保证新集群环境基线兼容统一，避免迁移后业务异常：

- 新集群系统版本：Ubuntu22\.04（与旧集群一致）

- K8s版本：v1\.32（统一集群版本，消除版本兼容问题）

- 运行时版本：containerd 版本与旧集群兼容对齐

- 网络插件：Calico版本统一，网段、IP池规划一致

- 存储插件：CSI存储驱动、存储类（StorageClass）名称与参数一致

- 内网Harbor镜像仓库连通正常，所有业务镜像预上传至新集群仓库

### 2\.2 迁移资源盘点

全面梳理旧集群所有业务资源，杜绝遗漏配置与数据：

```bash
# 全量导出集群资源清单
kubectl get all -o yaml --all-namespaces > old-cluster-all-resources.yaml

# 导出命名空间、存储、密钥、配置、权限完整资源
kubectl get ns -o yaml > old-ns.yaml
kubectl get sc,pv,pvc -o yaml > old-storage.yaml
kubectl get configmaps,secrets -o yaml --all-namespaces > old-config-secret.yaml
kubectl get ingress,service -o yaml --all-namespaces > old-network.yaml

```

### 2\.3 迁移前置备份（强制）

迁移属于高危变更，必须完成双重备份，保障故障可回滚：

- 执行 **ETCD 全量快照备份**，参考 `12-etcd-backup.md`

- 备份所有业务持久化数据、数据库数据、挂载存储数据

- 备份集群证书、kubeconfig、系统组件配置

### 2\.4 迁移窗口规划

- 配置与资源迁移：日间低影响时段分批操作

- 流量割接、核心数据同步：夜间业务低峰维护窗口执行

- 预留充足回滚时间，禁止临近业务高峰执行割接

## 3\. 新集群环境标准化搭建

严格按照生产基线搭建全新集群，保证环境纯净、标准化、无历史脏配置。

### 3\.1 集群基础搭建

- 部署 Ubuntu22\.04 系统，统一内核、系统参数、防火墙、时间同步基线

- 离线部署 containerd、kubeadm、kubelet、kubectl v1\.32 版本

- 初始化高可用K8s集群，配置内网Harbor镜像仓库

- 部署Calico网络插件、CSI存储插件、CoreDNS组件

### 3\.2 新集群健康验收

```bash
# 节点状态校验
kubectl get nodes
# 系统组件校验
kubectl get pods -n kube-system
# 网络存储校验

```

验收标准：所有节点Ready、组件Running、无报错、网络存储正常挂载。

### 3\.3 镜像离线同步

将旧集群所有业务镜像、系统镜像批量导出，上传至新集群内网Harbor，保证新集群可正常拉取镜像，规避离线环境镜像缺失问题。

## 4\. 集群资源批量迁移流程

遵循**底层资源优先、业务逐层迁移**顺序，禁止无序导入资源。

### 4\.1 第一步：迁移命名空间

```bash
kubectl apply -f old-ns.yaml

```

优先创建所有业务命名空间，保证后续资源导入目录正常。

### 4\.2 第二步：迁移配置与密钥

```bash
kubectl apply -f old-config-secret.yaml

```

迁移ConfigMap、Secret，保证业务启动依赖的配置、密钥完整。

### 4\.3 第三步：迁移存储资源

```bash
kubectl apply -f old-storage.yaml

```

同步StorageClass、PV、PVC，保证业务存储挂载链路正常，**有状态业务优先核对存储绑定关系**。

### 4\.4 第四步：迁移网络资源

```bash
kubectl apply -f old-network.yaml

```

同步Service、Ingress资源，预留流量入口，为后续割接做准备。

### 4\.5 第五步：迁移业务负载资源

```bash

```

无状态业务自动重建Pod，有状态业务需手动核对实例状态与数据挂载。

## 5\. 核心数据同步方案（有状态业务专属）

无状态业务迁移即可自愈，**数据库、缓存、中间件等有状态业务必须单独同步数据**。

### 5\.1 数据同步原则

- 采用「旧集群持续读写→全量数据备份→同步至新集群→增量数据追平」方案

- 数据同步期间旧集群持续对外提供服务，保证业务不中断

- 数据校验一致后，再执行流量割接

### 5\.2 持久化存储数据迁移

- 针对PV绑定的持久化数据，采用rsync离线同步数据文件

- 同步完成后校验文件完整性、权限一致性

- 新集群业务启动后核对数据完整性，杜绝数据丢失、错乱

## 6\. 灰度流量割接标准流程（零停机核心）

资源与数据同步完成后，执行**灰度流量切换**，逐步将业务流量从旧集群切换至新集群。

### 6\.1 割接前置校验

- 新集群所有业务Pod正常Running，无重启、崩溃、异常

- 新集群业务日志正常、接口可正常访问

- 新旧集群数据完全一致，无数据差异

- 监控指标平稳，无报错、无超时

### 6\.2 分阶段流量割接

1. **测试流量验证**：手动访问新集群业务接口，验证功能正常

2. **DNS灰度切换**：修改少量DNS解析，小流量切入新集群

3. **全量流量切换**：确认小流量无异常，全量切换业务流量至新集群

4. **旧集群保活观察**：流量切换后，旧集群持续保活观察24\~72小时

### 6\.3 割接异常回滚机制

流量切换后出现业务异常、数据错乱、报错飙升，立即**回切流量至旧集群**，终止新集群迁移，排查问题后重新迁移。

## 7\. 迁移后全维度验收标准

### 7\.1 资源层验收

- 新集群命名空间、配置、密钥、存储、网络资源完整对齐旧集群

- 所有业务负载实例数量、版本、配置与旧集群一致

### 7\.2 集群层验收

- 新集群节点全部Ready，系统组件运行稳定无异常

- 无Pending、CrashLoop、Error状态Pod

- 网络连通、存储挂载、DNS解析全部正常

### 7\.3 业务层验收

- 业务接口访问正常，QPS、延迟、错误率平稳无波动

- 所有业务功能正常，数据读写一致、无丢失、无错乱

- 业务日志无报错、无连接异常、无权限异常

## 8\. 旧集群下线销毁流程

新集群稳定运行、业务全量接管、观察期无异常后，方可执行旧集群下线操作。

### 8\.1 下线前置确认

- 业务流量100%运行在新集群，旧集群无实际业务流量

- 新旧数据同步完成，无数据滞后、差异

- 持续观察72小时，无任何业务异常、集群故障

### 8\.2 旧集群标准下线

严格遵循 `node-remove.md` 规范，逐台优雅下线、清空节点残留配置、重置集群环境，杜绝残留脏数据。

## 9\. 迁移常见问题与排错方案

- **业务Pod启动失败（镜像缺失）**：离线镜像未同步完整，补传对应镜像至新集群Harbor

- **存储挂载失败**：StorageClass、PV/PVC配置不匹配，核对存储插件与挂载参数

- **密钥配置失效**：Secret/ConfigMap未完整迁移，重新导入资源并重启业务Pod

- **有状态数据不一致**：数据同步不完整，重新全量\+增量同步数据，核对校验值

- **流量切换后报错**：Ingress/Service规则不一致，对齐新旧集群网络配置

## 10\. 生产迁移红线规范

- 禁止未搭建完成新集群、未校验基线直接迁移业务

- 禁止先销毁旧集群再迁移，无回滚兜底机制

- 禁止有状态业务不做数据同步、直接拉起业务

- 禁止一次性全量切流量，无灰度验证过程

- 禁止迁移过程不做备份、不留快照，故障无法回滚

- 禁止新集群未稳定验收，直接下线旧集群

- 禁止业务高峰期执行资源迁移、流量割接操作

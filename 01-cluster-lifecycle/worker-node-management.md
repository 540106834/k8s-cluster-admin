# Worker Node 节点管理规范（自建kubeadm \+ 云托管ACK/TKE/EKS 生产版）

## 1\. 文档概述

### 1\.1 文档定位

本文为 Kubernetes **Worker 节点全生命周期运维标准文档**，统一适配**kubeadm 自建集群**与 **ACK/TKE/EKS 云托管集群**。覆盖节点规划、初始化基线、标签污点管理、资源预留、日常巡检、业务调度管控、故障治理、节点分级规范，是生产集群节点运维的核心基准。

### 1\.2 核心职责边界

- **Worker 节点定位**：仅承载业务 Pod、Ingress、监控日志等数据面负载，不参与集群管控决策。

- **控制面不干涉**：Master/控制面仅负责调度与状态管理，不运行业务负载。

- **运维统一准则**：无论自建/云托管，Worker 节点运维、业务容错、资源治理均由运维团队自主负责。

## 2\. Worker 节点核心原理

### 2\.1 节点运行组件

所有 Worker 节点统一运行三大核心组件，是节点正常工作的基础：

- **kubelet**：节点代理，负责 Pod 生命周期管理、状态上报、健康检查、资源校验。

- **kube\-proxy**：维护集群 Service 转发规则，实现集群网络负载均衡。

- **containerd**：容器运行时，负责镜像拉取、容器启停、资源隔离。

云托管集群仅屏蔽控制面组件，Worker 节点组件运维逻辑与自建集群完全一致。

### 2\.2 节点状态机制

K8s 通过节点心跳机制判定节点状态，默认 40s 未上报标记为 **NotReady**，5m 未上报触发节点驱逐。

- **Ready**：节点健康，可正常调度业务 Pod

- **NotReady**：kubelet/网络/运行时异常，节点不可用

- **SchedulingDisabled**：节点被封锁，禁止新 Pod 调度

## 3\. 生产节点基线规范（所有节点统一落地）

适配 Ubuntu22\.04 \+ K8s v1\.32 生产环境，自建与云节点统一基线标准。

### 3\.1 系统强制基线

- 永久关闭 Swap、关闭冗余防火墙、开启 IP 转发

- 统一内核参数（bridge\-nf、tcp\_tw\_reuse、somaxconn）

- 时间同步正常，所有节点时间一致

- 文件句柄、进程数资源上限调高，适配高并发容器场景

- 运行时强制 systemd cgroup，杜绝 K8s v1\.32 版本兼容报错

### 3\.2 节点资源预留规范（生产必配）

防止节点资源被容器打满，导致节点雪崩、整机卡死，所有 Worker 节点必须配置 **kubelet 资源预留**。

- **系统预留**：CPU 200m、内存 256Mi、磁盘 10Gi

- **K8s组件预留**：CPU 100m、内存 256Mi

- **驱逐阈值**：内存低于10%、磁盘低于15%触发温和驱逐

云托管节点池可在平台界面直接配置，自建节点修改 kubelet config 固化。

## 4\. 节点标签与污点管理（业务隔离核心）

### 4\.1 节点标签（Label）业务分组

通过标签实现 **Pod 定向调度**，实现业务专属节点、环境隔离、机型隔离。

生产统一标签规范：

```bash
# 环境标签
kubectl label nodes node-xxx env=prod
# 业务线标签
kubectl label nodes node-xxx business=order
# 节点类型标签
kubectl label nodes node-xxx node-type=high-cpu
# 机房可用区标签
kubectl label nodes node-xxx az=az1
```

Pod 通过 `nodeSelector` 匹配对应节点，实现业务隔离混布管控。

### 4\.2 节点污点（Taint）调度限制

污点用于**禁止普通业务调度**，适配专属节点、独占节点、运维节点场景。

#### 4\.2\.1 污点三大策略

- **NoSchedule**：禁止新Pod调度，已有Pod不影响

- **PreferNoSchedule**：尽量不调度，软性限制

- **NoExecute**：禁止调度 \+ 驱逐已有Pod

#### 4\.2\.2 生产常用污点场景

```bash
# 设置专属节点，仅带容忍的Pod可调度
kubectl taint nodes node-xxx dedicated=middleware:NoSchedule

# 移除污点
kubectl taint nodes node-xxx dedicated:NoSchedule-
```

## 5\. 节点分层管理规范（生产强制）

禁止所有业务混跑同一节点池，严格按节点用途分层，与云集群节点池规范对齐。

### 5\.1 通用业务节点

- 承载常规微服务、无状态业务、后台任务

- 无特殊污点，默认可调度普通业务

### 5\.2 高性能计算节点

- 高CPU、高并发、接口网关业务

- 打标签 node\-type=high\-cpu，业务定向调度

### 5\.3 存储型节点

- 有状态服务、中间件、本地盘业务

- 专属污点隔离，避免普通业务抢占资源

### 5\.4 运维专属节点

- 运行 Ingress、Prometheus、日志、监控组件

- 污点隔离，禁止业务Pod抢占运维资源

## 6\. 节点日常运维操作规范

### 6\.1 节点信息查询

```bash
# 查看所有节点状态、角色、版本
kubectl get nodes -o wide

# 查看节点详细信息、污点、标签、资源分配
kubectl describe node node-xxx

# 查看节点资源占用情况
kubectl top node
```

### 6\.2 节点维护操作标准流程

节点升级、重启、补丁、硬件维护必须遵循：**封锁 → 驱逐 → 维护 → 恢复**

1. **封锁节点（Cordon）**：禁止新Pod调度

2. **驱逐节点（Drain）**：平滑迁移存量Pod至其他节点

3. **执行维护**：重启、升级、补丁、配置变更

4. **恢复节点（Uncordon）**：重新接纳调度

具体命令与详细场景参考文档：`node-maintenance.md`

## 7\. 自建 vs 云托管 Worker 节点运维差异

### 7\.1 自建 kubeadm 集群 Worker

- 完全自主运维系统、内核、运行时、kube组件

- 节点加入、移除、证书同步全手动操作

- 节点故障、组件异常需自主排查修复

- 资源预留、系统参数全自主配置

### 7\.2 云托管 ACK/TKE/EKS Worker

- 节点基于云厂商镜像初始化，基线统一

- 节点池支持批量扩容、缩容、滚动升级

- kubelet/containerd 版本由云平台统一管控

- 底层硬件故障、宿主机故障由厂商兜底替换

- 仅需维护业务层、标签污点、调度策略

## 8\. 节点常见故障与治理方案

### 8\.1 节点 NotReady 故障

**常见原因**：kubelet 异常、containerd 挂掉、网络插件异常、端口冲突、资源耗尽、系统卡死。

**排查顺序**：节点存活检测 → kubelet状态 → containerd状态 → 磁盘/内存负载 → CNI网络状态。

### 8\.2 节点负载过高

**治理方案**：

- 核查Pod资源限制，补齐 request/limit

- 打散密集业务至其他节点

- 扩容节点池，分散负载压力

- 优化节点资源预留，防止系统资源被抢占

### 8\.3 节点频繁重启、抖动

- 核查内核崩溃、OOM、磁盘只读

- 检查镜像异常、容器疯狂重启占用资源

- 核查云平台宿主机故障、硬件异常

## 9\. 生产红线禁止规范

- 禁止 Worker 节点不做资源预留直接上线业务

- 禁止核心业务单节点、单AZ部署，无冗余容错

- 禁止不同等级业务无隔离混布，引发资源抢占雪崩

- 禁止节点故障直接重启，不做驱逐、不做业务迁移

- 禁止随意修改节点标签、污点，导致业务调度错乱

- 禁止长期放任节点高负载、高重启、异常告警不处理

## 10\. 节点日常巡检清单

- 所有节点状态全部 Ready，无 NotReady、SchedulingDisabled

- 节点 CPU/内存/磁盘负载平稳，无长期高负载

- kubelet、containerd 无重启风暴、无报错日志

- 节点标签、污点与业务分层规范一致

- 资源预留、驱逐阈值配置合规生效

- 跨AZ部署正常，无单点集中部署风险

# 集群规划指南（生产实战版）

## 1. 文档说明

### 1.1 定位

本文为 **生产级 Kubernetes 集群上线前标准化规划方案**，是集群部署、扩容、运维、迁移的前置基准文档。所有后续安装、升级、运维操作均以本文规划基线为标准，杜绝随意部署、参数混乱、架构不统一导致的后期重构与故障。  
本文覆盖：架构选型、节点规划、IP 地址规划、网络规划、版本选型、资源配额、高可用方案、安全基线、扩容与容灾预案、生产禁忌清单。

### 1.2 适用场景

- 企业自建生产 K8s 集群（裸金属 / 虚拟机）
- 基于 kubeadm / RKE2 部署的标准集群
- 需要长期迭代、稳定运行、可运维、可迁移的业务集群
- 多环境统一规范（测试、预发、生产对齐基线）

### 1.3 前置原则（生产核心准则）

- **可观测**：所有组件可监控、可告警、可审计
- **高可用** ：控制面多副本、核心组件无单点、etcd 集群高可用
- **可灰度**：升级、变更、扩容支持滚动、分批、回滚
- **可灾备**：etcd 定时备份、数据持久化、故障可恢复
- **规范统一**：主机名、IP、网段、标签、污点、资源规格统一标准化

## 2. 集群架构选型规划

### 2.1 架构模式选择（生产唯一推荐）

生产环境**禁止单 Master 架构**，强制采用：**3 Master + N Worker 高可用架构**
- 控制面：3 节点（容忍 1 台 Master 宕机不影响集群可用）
- etcd：3 节点堆叠部署（与 Master 共节点，生产标准最简高可用架构）
- Worker 节点：根据业务负载横向扩容
- VIP 虚拟 IP：用于 apiserver 负载均衡，统一集群接入入口

### 2.2 部署工具选型规范

|部署工具|适用场景|生产优势|约束条件|
|---|---|---|---|
|kubeadm|中小规模集群、高度自定义、自研底座|轻量化、可控性强、组件解耦、便于深度运维调优|需自行维护运行时、证书、高可用负载均衡|
|RKE2|大规模集群、标准化交付、合规场景|一体化交付、自动证书轮换、内置运维能力、批量部署友好|标准化程度高，自定义改造成本略高|

### 2.3 集群环境隔离规划

严格环境隔离，**禁止测试/预发/生产混用集群**：
- DEV 开发集群：供开发自测、功能验证，资源可超配
- FAT 测试集群：流水线自动化测试、版本验证
- UAT 预发集群：与生产架构一致，用于上线前验证、压测
- PROD 生产集群：业务正式运行，严格资源管控、变更管控

### 2.4 云托管 K8s 集群差异化说明（ACK/TKE/EKS）

本文主体默认基于**自建裸金属/虚拟机 K8s 集群（kubeadm/RKE2）**规划，云托管 K8s 为公有云托管形态，架构、运维、规划规则与自建集群存在核心差异，本章节统一标注全量差异化特性，后续所有规划条款均适配双场景。
- **控制面托管差异**：云 K8s 控制面（apiserver/etcd/controller）由云厂商全权托管，默认多副本高可用，无需自行规划 3Master 架构、无需维护 VIP、无需手动搭建 etcd 集群，彻底免除控制面运维、证书轮换、etcd 灾备工作。
- **架构约束差异**：云 K8s 不支持自定义控制面节点规格、不允许登录 Master 节点、禁止修改控制面底层配置，架构标准化程度极高，自定义改造能力弱于自建集群。
- **扩容能力差异**：云 K8s 工作节点支持弹性伸缩、节点池批量管理、按需付费，无需提前规划硬件资源，适配业务突发流量；自建集群需提前规划硬件、预留扩容资源。
- **运维职责差异**：云厂商负责集群控制面稳定性、版本迭代、安全补丁、证书自动续期；用户仅负责 Worker 节点、业务应用、存储、网络策略运维。

严格环境隔离，**禁止测试/预发/生产混用集群**：
- DEV 开发集群：供开发自测、功能验证，资源可超配
- FAT 测试集群：流水线自动化测试、版本验证
- UAT 预发集群：与生产架构一致，用于上线前验证、压测
- PROD 生产集群：业务正式运行，严格资源管控、变更管控

## 3. 节点硬件与系统规划

### 3.1 节点角色规格基线（生产标准）

#### 3.1.1 Master 节点（控制面）

核心原则：**控制面不跑业务负载，资源独占，保证稳定性**
- 配置基线：4C8G / 8C16G（推荐 8C16G 及以上）
- 磁盘：系统盘 SSD > 100G，etcd 数据盘独立 SSD（高性能优先）
- 数量：固定 3 台，不扩容、不缩容
- 调度约束：默认打污点，禁止普通业务 Pod 调度至 Master

**【云托管 K8s 差异注释】**：ACK/TKE/EKS 无 Master 节点概念，用户无需规划控制面硬件、磁盘、数量及污点策略，所有控制面运维由云厂商托管，仅需关注工作节点池配置。

#### 3.1.2 Worker 节点（业务面）

- 配置基线：8C16G / 16C32G / 32C64G（根据业务分级适配）
- 磁盘：系统盘 SSD，业务数据盘独立分区/云盘
- 数量：按需横向扩容，支持分批灰度扩容

**【云托管 K8s 差异注释】**：云 K8s 工作节点基于云服务器，支持按需配置规格、磁盘类型（高效云盘/SSD/极致云盘），支持节点池分组管理，可按业务场景配置弹性伸缩规则，无需手动规划固定节点数量。

### 3.2 操作系统基线

生产统一操作系统，避免系统混杂导致兼容性问题：
- 推荐：CentOS 7.9 / Ubuntu 20.04 / 22.04（长期支持版）
- 内核要求：>= 5.4（推荐 5.15+，适配新特性、网络优化）
- 系统初始化：关闭防火墙/配置放行策略、关闭SELinux、关闭swap、时间同步、内核参数调优

**【云托管 K8s 差异注释】**：云 K8s 节点为厂商定制镜像，默认已完成内核调优、Swap 关闭、SELinux 关闭、网络参数优化等基线配置，用户无需手动初始化系统，仅需按需配置节点镜像版本。

### 3.3 节点命名规范

统一主机名规范，便于运维识别、监控标签、日志归集：
- Master：prod-k8s-master-01/02/03
- Worker：prod-k8s-worker-01/02/03...

**【云托管 K8s 差异注释】**：云 K8s 节点主机名由平台自动生成，不支持自定义固定命名，可通过自定义节点标签、节点池名称实现业务分组运维，替代主机名识别方案。

## 4. 网络与 IP 规划（生产核心）

### 4.1 网络网段整体划分

集群网络分为四大网段，**严格网段隔离，不与主机网段冲突**：
1. **主机物理网段**：服务器物理 IP，用于 SSH、运维、监控、内网通信
2. **Pod 网段**：集群 Pod 内部通信地址段
3. **Service 网段**：集群内部服务虚拟 IP 段
4. **Ingress/外网网段**：业务对外暴露网段

**【云托管 K8s 差异注释】**：云 K8s 网络深度对接公有云 VPC 体系，Pod/Service 网段需在 VPC 私网网段内规划，禁止与 VPC 子网冲突；外网暴露统一对接云厂商 SLB，无需手动规划节点外网网段，支持内网 SLB/公网 SLB 一键切换。

### 4.2 推荐标准网段配置

为避免后期网段不足、冲突、跨环境互通问题，统一规划：
- Pod CIDR：`10.244.0.0/16`（默认足够支撑万级 Pod）
- Service CIDR：`10.96.0.0/16`
- DNS 服务固定 IP：`10.96.0.10`
- APIServer VIP：内网闲置固定虚拟 IP（同主机网段）

**【云托管 K8s 差异注释】**：云 K8s 不支持自定义 APIServer VIP，控制面入口为厂商自动分配的域名/公网 IP；Pod/Service 网段需在创建集群时指定，创建后**不可修改**，需提前精准规划网段容量，避免后期扩容网段不足。

### 4.3 CNI 网络插件选型

|CNI 插件|生产适用场景|优缺点|
|---|---|---|
|Flannel|中小集群、简单稳定、运维成本低|部署简单、稳定可靠，无复杂策略，性能中等|
|Calico|中大型集群、需要网络策略、安全隔离|支持 NetworkPolicy、网络隔离、带宽管控，生产首选|

**生产推荐：Calico**，满足微服务隔离、安全访问控制、合规要求。

**【云托管 K8s 差异注释】**：主流云 K8s（ACK/TKE/EKS）默认搭载厂商优化 CNI（如阿里云 Terway、腾讯云 VPC-CNI），优先使用云原生 CNI，支持 Pod 独占网卡、固定 IP、VPC 直通，网络性能优于 Calico/Flannel，且天然适配云防火墙、安全组、网络ACL，无需自行维护 CNI 组件。

## 5. Kubernetes 版本选型规划

### 5.1 版本选择原则

- 选择 **官方稳定版、长期维护版**，不追最新灰度版
- 生产集群统一版本，禁止多版本混杂
- 预留 3 个月以上兼容缓冲，避免快速迭代带来兼容性问题

**【云托管 K8s 差异注释】**：云 K8s 仅支持平台提供的认证稳定版本，版本迭代由云厂商统一灰度推进，用户无法自定义编译、降级内核版本；版本升级需遵循厂商迭代节奏，仅支持跨指定小版本升级，禁止自定义跳跃升级。

### 5.2 生产推荐版本

当前生产稳定基线：**v1.27 / v1.28 / v1.29**
- 摒弃老旧版本：<=1.24（大量资源 API 废弃、证书机制老旧）
- 规避极新版本：v1.30+ 新特性未充分生产验证

**【云托管 K8s 差异注释】**：云厂商会主动下线停止维护的老旧 K8s 版本，需提前关注厂商版本生命周期公告，按时灰度升级，避免版本过期被强制升级；极新版本通常延迟1-2个季度上线，保证生产稳定性。

### 5.3 版本约束规范

- 控制面与节点版本小版本允许偏差（不超过 1 个小版本）
- 集群内部所有组件 kubelet/kube-proxy 版本统一对齐
- 后续升级遵循：**小版本迭代、季度灰度升级**，禁止跨多版本跳跃升级

**【云托管 K8s 差异注释】**：云 K8s 控制面升级由厂商全自动完成，用户仅需升级工作节点版本，平台会自动适配版本偏差规则，杜绝版本不兼容问题；同时支持节点池分批灰度升级，业务无感知。

## 6. 资源调度与配额规划

### 6.1 节点资源预留（生产必配）

为防止节点资源打满导致集群雪崩，强制配置系统预留资源：
- kubelet 预留：CPU、内存、磁盘资源，保障系统与集群组件优先调度
- 节点最大 Pod 数限制：默认 110，高密度场景可按需调整，避免端口耗尽

**【云托管 K8s 差异注释】**：云 K8s 节点资源预留由平台自动配置，默认预留系统、kubelet 资源，无需手动修改；节点最大 Pod 数由节点规格、CNI 类型限制，固定阈值，不可自定义调整。

### 6.2 命名空间资源配额规划

生产集群必须开启 Namespace 资源配额，防止单业务抢占全集群资源：
- CPU / Memory 总配额
- Pod 最大数量限制
- PVC、ConfigMap、Secret 资源数量限制

### 6.3 节点标签与污点规划

- Master 节点：污点隔离，禁止业务调度
- 异构节点：按标签区分高CPU、高内存、GPU节点，实现业务亲和调度
- 环境标签：区分测试/生产节点，防止跨环境调度错误

**【云托管 K8s 差异注释】**：云 K8s 控制面无节点污点配置，工作节点池支持自定义标签、污点、亲和策略，同时支持节点池分级调度，可精准区分通用业务、GPU 业务、高负载业务节点，调度策略更灵活。

## 7. 高可用与容灾规划

### 7.1 控制面高可用

- 3 台 Master 冗余，单 Master 宕机不影响集群可用性
- APIServer 通过 VIP 负载均衡，自动剔除故障节点
